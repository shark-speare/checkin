# Discord 機器人打卡系統
1627 32 趙崇崴

利用 Discord 機器人配合開發板實作打卡管理系統

# API

* 可接收 `/new/{name}/{rfid}` 來註冊員工
* 可接收 `/check/{rfid}` 來打卡
* 自動判斷上下班

此部分使用開發板簡單發出API請求即可，資料庫的操作交由伺服器端

# Discord 指令

## 員工

### /staff remove
移除員工

### /staff check
取得目前打卡上班/下班的員工


## 打卡紀錄

### /record all
取得全部打卡紀錄

### /record today
取得今日全部打卡紀錄

### /record staff
取得該員工打卡紀錄表

# 專案開發心得

本次作業使用AI輔助學習，但**AI生成的程式碼佔比為0%**，是這個時代難得一見的純手工code

## 三層架構
我以往做專案都很隨性(見之前幫社團寫的機器人[hsnucomputer](https://github.com/shark-speare/hsnucomputer))，各種檔案階層全部塞在主函數裡面；

但是近期學到了簡單的repo/service/main層的概念，這三層架構的特點就是把關注點分開，每一層不應該知道上一層的細節，詳細的動作由每一層各自封裝

* repository: 資料庫CRUD操作(實際SQL語句)
* service: 處理業務邏輯(資料清洗、資料合法性)，呼叫repo層的函數
* main: service的入口層

所以這次我跟以往的專案非常不同，嘗試把這三層架構實作出來，並且用model封裝員工和紀錄資料

## Database物件
這次我還學到關於類別專用實例的概念

sqlite不像其他資料庫有連線池，而是檔案形式的資料庫，每次存取資料庫都會上Lock，所以開很多連線是沒有意義的

這次我嘗試使用Database的類別屬性，以及__new__()魔法函數，在資料庫連線物件建立前攔截下來確保不會多開連線，如果開重複了就使用原本的連線實例

## API
這次也去碰了一點 FastAPI，因為不想讓開發版端的邏輯太複雜，所以設計了呼叫接口，這樣開發板和伺服器可以在異地溝通，也方便部署多個開發板在不同公司(或是工作室、學校)出入口

## 資料庫
因為手邊剛好也有在開發跟關聯式資料庫有關係的專案，所以這段期間多學了很多關聯式資料庫的概念和知識點

本次著手開發以前的能力範圍大概只有SELECT/INSERT/CREATE TABLE，而且一個檔案只會開一張表

但這次因為要管理員工的RFID，也要管理紀錄對應員工，所以去學習了FOREIGN KEY的用法。

在比對資料求當前簽到/簽退的員工時也使用到了JOIN。我原本的想法是，我已經做好生成Staff物件的函數了，不如我就每個員工都取出來，再篩選誰已經簽到/退了，但是我嘗試把這部分的邏輯讓sqlite來運算，往後面對資料量更大的時候才不會爆掉
